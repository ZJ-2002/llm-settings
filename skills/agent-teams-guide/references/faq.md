## 一、团队规模问题

### Q1: Team 适合什么规模的任务？

**A**:
| 文件数 | 推荐配置 | 预计时长 |
|--------|----------|----------|
| 1-2 个 | 单人，不用 Team | 5-10 分钟 |
| 3-5 个 | 2 人 Team | 10-20 分钟 |
| 5-10 个 | 3 人 Team | 15-30 分钟 |
| 10+ 个 | 4-5 人 Team | 30-60 分钟 |

**经验法则**：
- < 100 行代码：单人
- 100-500 行：2 人
- > 500 行：3+ 人

### Q2: 什么时候应该增加成员？

**A**: **不要提前生成成员**。遵循以下原则：
1. 先分析任务复杂度
2. 创建任务列表
3. 按需生成成员（只在成员即将有工作时）
4. 避免成员空闲等待

### Q3: 最多可以用多少个成员？

**A**: 理论上无限制，但**建议不超过 5 人**：
- 成员越多，通信开销越大
- 协调复杂度指数增长
- 收益递减

## 二、任务管理问题

### Q4: 任务粒度应该如何控制？

**A**: **每个任务应该能在单次会话内完成**。

**好的任务**：
```
- 实现用户注册 API（1 个文件，~100 行）
- 编写登录测试（1 个文件，~50 行）
```

**不好的任务**：
```
- 实现完整的用户认证系统（太大，应该拆分）
```

### Q5: 依赖关系如何设置？

**A**: 使用 `addBlockedBy` 设置依赖：
```bash
TaskUpdate(taskId="3", addBlockedBy=["1", "2"])  # 任务3依赖任务1和2
```

**原则**：
- 只在必要时设置依赖
- 避免循环依赖
- 保持依赖链尽可能短

### Q6: 任务失败了怎么办？

**A**: 自动重试机制：
1. 失败后，任务状态保持 `failed`
2. `retry_count` 自动 +1
3. 如果 `retry_count < 3`，可以重置为 `pending` 重试
4. 如果达到 3 次，需要人工介入

## 三、通信问题

### Q7: 成员 idle 了怎么办？

**A**: **这是正常的！**
- 成员完成工作后会自动进入 idle 状态
- 发送 SendMessage 即可唤醒他们
- **不要把 idle 当做错误**

### Q8: 什么时候用 broadcast？

**A**: **90% 的情况用 message**。只在以下情况用 broadcast：
1. 紧急阻塞（影响所有人的问题）
2. 重大方向变更
3. 全员收尾通知

**错误做法**：
```bash
# 只需要通知一个人却用广播
SendMessage(type="broadcast", content="developer，你的任务#1完成了吗？")
```

### Q9: Leader 需要轮询成员状态吗？

**A**: **不需要！**
- 消息会自动送达
- 等待成员的 SendMessage 通知即可
- 轮询会浪费大量 API 调用

## 四、性能问题

### Q10: 如何降低成本？

**A**: 5 个优化方向：
1. **减少成员数**：从 2 人开始，必要时再增加（节省 30-50%）
2. **避免循环**：一次修复成功（节省 40%）
3. **减少通信**：合并消息（节省 20%）
4. **选择模型**：非核心任务用 Haiku（节省 60-70%）
5. **避免 broadcast**：用 message（节省 25% × N）

### Q11: 如何提高效率？

**A**:
1. **按需生成成员**：不要提前生成
2. **批量操作**：合并多个更新
3. **并行执行**：设置独立任务并行
4. **合理依赖**：避免不必要的阻塞
5. **及时关闭**：完成后立即关闭团队

## 五、错误排查问题

### Q12: 成员长时间无响应怎么办？

**A**: 诊断步骤：
1. 发送测试消息：`SendMessage(content="ping")`
2. 检查日志：`tail -50 ~/.claude/teams/{team-name}/.state/session-log.md`
3. 如果仍无响应，重新生成成员

### Q13: 任务一直被阻塞怎么办？

**A**: 检查依赖关系：
```bash
# 查看任务的 blockedBy
TaskGet(taskId="3")
# 检查前置任务状态
TaskList()
```

可能的原因：
- 前置任务未完成
- 依赖关系错误
- 循环依赖

### Q14: 广播消息没人收到怎么办？

**A**: 检查：
1. 是否有活跃成员（TaskList）
2. 成员是否已关闭
3. 网络是否正常

### Q15: 如何查看错误日志？

**A**:
```bash
# 查看最近错误
grep -i "error\|fail" ~/.claude/teams/{team-name}/.state/session-log.md | tail -10

# 查看特定任务
grep "task-001" ~/.claude/teams/{team-name}/.state/session-log.md
```

## 六、最佳实践

### Q16: 新项目如何开始？

**A**: 推荐流程：
1. 创建团队：`TeamCreate()`
2. 创建任务：`TaskCreate()` × N
3. 设置依赖：`TaskUpdate(addBlockedBy=[...])`
4. 生成成员：`Task()` × M
5. 分配任务：`TaskUpdate(owner="...")`
6. 发送消息：`SendMessage()`
7. 等待完成...
8. 关闭团队：`TeamDelete()`

### Q17: 如何选择 subagent_type？

**A**:
| 任务类型 | subagent_type | 原因 |
|----------|---------------|------|
| 开发、测试 | `general-purpose` | 需要读写执行 |
| 代码审查 | `Explore` | 只读即可 |
| 架构设计 | `Plan` | 只读即可 |
| 运维操作 | `Bash` | 仅命令执行 |

### Q18: 如何避免反模式？

**A**: 避免以下行为：
- ❌ Leader 轮询成员状态
- ❌ 成员完成不通知 Leader
- ❌ Leader 亲自写代码
- ❌ 无视依赖直接开始
- ❌ 滥用 broadcast
- ❌ 消息不包含文件信息

详见 [anti-patterns.md](./anti-patterns.md)

## 七、故障恢复

### Q19: 项目中断了怎么恢复？

**A**:
1. 运行恢复脚本：
   ```bash
   python3 ~/.claude/mcp-servers/agent-teams-mcp/diagnose.py diagnose {team-name}
   ```
2. 检查 in_progress 任务
3. 检查 Git 状态
4. 继续执行

### Q20: 如何清理残留团队？

**A**:
```bash
# 查看团队配置
ls ~/.claude/teams/

# 确认无活跃成员后删除
rm -rf ~/.claude/teams/{team-name}
```
